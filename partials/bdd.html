Behavior-Driven Development (BDD)

What is BDD?

BDD is a second-generation, outside-in, pull-based, multiple-stakeholder, multiple-scale, high-automation, agile methodology. It describes a cycle of interactions with well-defined outputs, resulting in the delivery of working, tested software that matters.

BDD was created by Dan North as as a response to the issues encountered teaching test-driven development (TDD) and is basically a way to rethink the approach to unit testing and acceptance testing.

For more information about BDD, please check the wikipedia or Dan North's introduction about BDD in his web page.

Brief explanation of how does BDD work

BDD stories consist basically of a title, the narrative and the acceptance criteria.

  • Title: This is typically one line representing the story.
  • Narrative: It tries to specify a feature, a role and a benefit by using a template such as: As a "role" I want to "feature" so that "benefit"
  • Acceptance Criteria: The acceptance criteria tries to define under which conditions the story is considered to be met. It consists of one or more scenarios that define this criteria.

For more detailed information about BDD stories, please check Dan North's section what is a story
